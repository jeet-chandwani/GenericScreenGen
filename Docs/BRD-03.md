# This file has my requirements for adding more layout policies and field types.

**Req 3.1.** Add new layout policy = tabular used for displaying multiple records in a table format. Detail actions and features are describved below for this layout policy:
    **Req 3.1.0**. We should add a storage abstraction `IDataStore` for persisted row data operations and implement an initial JSON-backed data store (`CJsonDataStore`) under solution folder/project area `MyJsonDataStore`. The design should be pluggable so we can later swap the underlying store implementation (for example, database-backed) with minimal changes.
    **Req 3.1.1**. All fields should be displayed as one record per row, like cells in a table. The field label should be displayed as a header for each column and the corresponding controls should be displayed below the headers. The layout should adjust dynamically as screen size changes.
    **Req 3.1.2**. We should support vertical scroll as needed based upon # of rows. The headers should be fixed and should not scroll with the fields. 
    **Req 3.1.3**. We should support horizontal scroll as needed based upon # of columns and screen size. 
    **Req 3.1.4**. We should support sorting of rows based upon any column. The sorting should be done in ascending or descending order when user clicks on the column header. The sorted column and sort order should be visually indicated in the UI.
    **Req 3.1.5**. We should support filtering of rows based upon any column. The filtering should be done by providing a filter input box for each column header. When user enters a value in the filter input box, the rows should be filtered to show only those rows that match the filter criteria for that column. The filtering should be case-insensitive and should support partial matches.
    **Req 3.1.6**. We should support filters that allow user to filter rows based upon multiple columns at the same time. The filters should work together to narrow down the results based upon the criteria specified in each filter input box. For example, if user enters a value in the filter input box for column A and another value in the filter input box for column B, the rows should be filtered to show only those rows that match both criteria for column A and column B.
    **Req 3.1.7**. We should support pagination of rows if there are more than 50 rows. The pagination controls should allow user to navigate to next page, previous page, first page, and last page. The pagination controls should also display the current page number and total number of pages.
    **Req 3.1.8**. We should support exporting of data to CSV format. The export functionality should allow user to export all rows or only the filtered rows based upon the current filter criteria. The exported CSV file should include the column headers and the corresponding data for each row.
    **Req 3.1.9**.  We should support editing of data in a separate edit screen. When user clicks on a row, it should navigate to a separate screen where user can edit the data for that row, which may typically follow the record-detail layout and follow the layout policy for that page accordingly for edit/display and cancel/save.
    **Req 3.1.10**. We should support adding new rows of data. There should be an "Add New Row" button that allows user to add a new row of data. When user clicks on the "Add New Row" button, it should navigate to a separate screen where user can enter the data for the new row. The new row screen should display the field labels and corresponding controls for each column in that row. The new row screen should also have a save button to save the new row and a cancel button to discard the new row and navigate back to the tabular view.
    **Req 3.1.11**. We should support deleting of rows. There should be a delete button for each row that allows user to delete that row of data. When user clicks on the delete button, it should prompt the user to confirm the deletion. If user confirms the deletion, the row should be deleted from the data and the tabular view should be updated accordingly.
**Req 3.2** Add new layout policy = record-detail used to display and edit of a single record at a time. 
    **Req 3.2.1** This layout policy will be similar to per-line layout policy for now, more refinements will be added later.
    **Req 3.2.2** We should have a save button to save the changes and a cancel button to discard the changes and navigate back to the tabular view.
    **Req 3.2.3** We should support showing original values. When user clicks on a "Show Original Values" button, it should display the original values of the fields before any changes were made. This will allow user to compare the current values with the original values and make informed decisions about saving or discarding changes.

**Req 3.3** Refine the lookup field type to support if mandatory, optional, single and multiple selection.
    **Req 3.3.1** The lookup field should also support search functionality to allow user to easily find the desired value from the list of options. The search functionality should be case-insensitive and should support partial matches. 
    **Req 3.3.2** The lookup field should also support displaying additional information about each option, such as a description or an image, to help user make informed decisions when selecting an option.
    **Req 3.3.3** For multiple selection lookup field, the selected values should be displayed in a clear and organized manner, such as using tags or a multi-select dropdown. The user should also have the ability to easily remove selected values if needed.
